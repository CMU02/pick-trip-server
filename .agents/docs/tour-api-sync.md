# TourAPI 수집·동기화 전략

이 문서는 PickTrip 서버가 공공데이터 포털 OpenAPI에서 콘텐츠를 수집하고 동기화하는 전략을 정의한다.

## 사용 API

- `국문 관광정보 서비스_GW`: 지역 콘텐츠, 행사, 상세정보, 이미지, 코드, 동기화의 1차 원천
- `관광사진 정보_GW`: 콘텐츠 대표 이미지가 부족할 때 보조 이미지 원천

인증키 주입 방법은 `.agents/docs/package-manager-guide.md` 참고.

## API 공통 규칙

- 응답 형식은 JSON: `_type=json`
- 공통 요청 파라미터: `serviceKey`, `MobileOS=ETC`, `MobileApp=PickTrip`, `pageNo`, `numOfRows`
- 개발 계정 트래픽은 일 1,000건 기준. 초기 수집은 배치와 캐시를 전제로 한다.

## 초기 수집 전략

### 1단계 — 지역 코드 수집

1. `/areaCode2`로 경상남도, 경상북도 코드 확인
2. 각 시도 코드로 하동군, 영주시, 예천군 `sigunguCode` 확인
3. `/ldongCode2`로 법정동 코드 함께 저장
4. `regions` 테이블에 내부 코드(`HADONG`, `YEONGJU`, `YECHEON`)와 매핑

### 2단계 — 분류 코드 수집

1. `/categoryCode2`로 관광 타입 코드 확인
2. `/lclsSystmCode2`로 신분류체계 1/2/3Depth 코드 수집
3. 신분류체계 정의서(`lclsSystm` XLSX) 기반으로 PickTrip 내부 카테고리와 매핑

### 3단계 — 콘텐츠 수집

1. 지역별 `/areaBasedList2` 호출
2. MVP 대상 `contentTypeId`만 필터링: `12`, `14`, `15`, `25`, `28`, `38`, `39`
3. 각 콘텐츠별 `/detailCommon2` 호출
4. 각 콘텐츠별 `/detailIntro2` 호출
5. 필요한 경우 `/detailInfo2` 호출
6. 각 콘텐츠별 `/detailImage2` 호출
7. 이미지가 부족하면 `/gallerySearchList1`을 장소명+지역명 조합으로 보조 호출

### 4단계 — 축제 데이터 수집

- `/searchFestival2`는 여행 날짜 기반으로 호출한다.
- `eventStartDate`, `eventEndDate`를 AI 일정 생성 제약 조건으로 저장한다.
- 여행 날짜와 맞지 않는 축제는 목록에는 노출하되 AI 일정 생성 후보에서는 제외하거나 "기간 불일치"로 표시한다.

## 검색 전략

- 일반 목록: DB에 적재된 콘텐츠를 우선 검색한다.
- DB에 충분하지 않은 경우 `/searchKeyword2`를 보조 호출한다.
- 외부 API 직접 호출 결과는 즉시 노출하지 않고 내부 저장/정규화 후 반환한다.

## 동기화 전략

### 국문 관광정보

- `/areaBasedSyncList2`로 변경 콘텐츠를 감지한다.
- 수정된 콘텐츠는 `contentid` 기준으로 상세 API를 재호출한다.
- 삭제 또는 비표시 콘텐츠는 `dataStatus`를 `INACTIVE` 또는 `DELETED`로 변경한다.
- 운영시간, 휴무일, 축제 기간은 변동성이 높으므로 `dataVerifiedAt`을 별도로 관리한다.

### 관광사진

- `/gallerySyncDetailList1`에서 `galUseFlag=1`인 사진만 사용한다.
- `galUseFlag=0` 또는 비표시 상태는 이미지 후보에서 제외한다.
- 관광사진은 콘텐츠 원천 ID와 직접 연결되지 않을 수 있으므로 `title`, `regionName` 기반으로 매칭하고 검수 상태를 관리한다.

## 자체 보완이 필요한 필드

TourAPI만으로는 가족 여행 품질 기준을 충족하기 어려운 필드는 운영자 검수 또는 별도 관리로 보완한다.

- 아이 동반 적합도
- 부모님 동반 적합도
- 걷기 부담
- 화장실 편의성
- 실내/실외 구분
- 비 오는 날 대안 가능 여부
- 예약 필요 여부
- 예상 체류 시간
- AI 일정 생성 시 우선 배치/제외 사유
