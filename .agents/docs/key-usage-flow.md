# 주요 사용 흐름

이 문서는 사용자의 앱 사용 흐름을 Spring 서버 API 관점으로 재구성한다. 실제 엔드포인트 이름은 구현 단계에서 OpenAPI 문서로 확정하되, 아래 흐름과 책임은 유지한다.

## 1. 로그인 / 회원가입

1. 클라이언트가 Kakao 또는 Google 인증을 완료하고 인가 코드 또는 제공자 토큰을 서버로 전달한다.
2. 서버는 외부 OAuth 제공자와 통신해 사용자 식별 정보를 검증한다.
3. 서버는 내부 사용자 계정을 조회하거나 생성한다.
4. 서버는 인증 토큰을 발급한다.
5. 클라이언트는 이후 인증이 필요한 API에 토큰을 포함한다.

비로그인 사용자는 콘텐츠 탐색 API를 사용할 수 있다. 여행 바구니 저장, 일정 저장, 일정 공유는 로그인이 필요하다.

## 2. 여행 조건 설정

1. 사용자가 지역을 선택한다.
   - 하동
   - 영주
   - 예천
2. 사용자가 여행 날짜와 기간을 선택한다.
   - 당일치기
   - 1박 2일
   - 2박 3일
   - 커스텀
3. 사용자가 동행 조건을 선택한다.
   - 아이와 함께
   - 부모님과 함께
   - 가족 전체
   - 걷기 적게
   - 자연 위주
   - 체험 위주
   - 음식 위주
   - 실내 대안 필요

서버는 이 조건을 일정 생성 요청의 입력값으로 사용한다. 로그인 사용자의 경우 조건을 임시 여행 바구니나 저장 전 일정 초안에 연결할 수 있다.

## 3. 지역 콘텐츠 탐색

1. 클라이언트가 지역, 카테고리, 검색어, 필터 조건을 서버에 전달한다.
2. 서버는 조건에 맞는 콘텐츠 목록을 조회한다.
3. 서버는 콘텐츠 카드에 필요한 요약 정보를 반환한다.
4. 사용자가 특정 콘텐츠를 선택하면 서버는 상세 정보를 반환한다.

지원 카테고리는 다음과 같다.

- 음식
- 축제 및 행사
- 자연 관광지
- 문화 및 역사
- 체험
- 시장 및 로컬 상권
- 비 오는 날 갈 곳

## 4. 여행 바구니 구성

1. 사용자가 콘텐츠를 바구니에 담는다.
2. 서버는 사용자, 여행 조건, 콘텐츠, 우선순위를 연결해 저장한다.
3. 사용자는 각 콘텐츠의 우선순위를 변경할 수 있다.

우선순위는 다음 세 가지로 제한한다.

- `MUST_VISIT`: 꼭 가기
- `PREFERRED`: 가면 좋음
- `OPTIONAL`: 시간 남으면

## 5. AI 일정 생성

1. 클라이언트가 여행 바구니와 여행 조건을 기준으로 일정 생성을 요청한다.
2. 서버는 최소 입력 조건을 검증한다.
   - 지역이 선택되어 있어야 한다.
   - 여행 날짜 또는 기간이 있어야 한다.
   - 콘텐츠가 2개 이상 선택되어 있어야 한다.
3. 서버는 콘텐츠 운영시간, 축제 기간, 체류 시간, 동행 조건을 AI 입력으로 변환한다.
4. 서버는 AI 응답을 검증해 일정 구조로 변환한다.
5. 서버는 일차별 일정, 장소 목록, 배치 이유를 반환한다.

## 6. 일정 수정

사용자는 생성된 일정을 확인한 뒤 다음 작업을 수행할 수 있다.

- 장소 순서 변경
- 장소 삭제
- 대체 장소 추가
- 식사 시간 변경
- 특정 장소 고정
- 전체 일정 다시 생성
- 하루 단위 다시 생성

서버는 수정된 결과가 저장 가능한 일정 구조인지 검증한다.

## 7. 일정 저장

1. 클라이언트가 최종 일정을 저장 요청한다.
2. 서버는 인증 사용자인지 확인한다.
3. 서버는 일정, 일차, 장소, 배치 이유를 저장한다.
4. 서버는 저장된 일정 ID와 마지막 수정 시각을 반환한다.

## 8. 일정 공유

1. 사용자가 저장된 일정의 공유 링크 생성을 요청한다.
2. 서버는 일정 소유자를 검증한다.
3. 서버는 예측 불가능한 공유 토큰을 생성한다.
4. 클라이언트는 공유 URL을 외부로 전달한다.
5. 공유 URL 접근자는 인증 없이 공유용 일정 조회 API를 사용할 수 있다.
