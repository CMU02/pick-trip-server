# 정보 보안 및 데이터 신뢰성 보완 방안

PickTrip 서버는 인증 정보, 사용자 일정, 외부 API 데이터, AI 생성 결과를 함께 다룬다. 따라서 보안과 데이터 신뢰성을 기능 설계 단계부터 분리해서 관리해야 한다.

## 1. 인증과 인가

- Kakao, Google OAuth 연동 시 외부 제공자 토큰을 서버에서 검증한다.
- 내부 사용자 ID와 외부 제공자 사용자 ID를 분리해서 저장한다.
- 인증이 필요한 API와 공개 API를 명확히 분리한다.
- 다른 사용자의 여행 바구니, 저장 일정, 공유 설정에 접근할 수 없도록 소유자 검증을 수행한다.
- 공유 링크 조회는 별도 권한 정책을 사용하고, 내부 일정 ID를 직접 노출하지 않는다.

## 2. 토큰과 세션

- 액세스 토큰 만료 시간을 짧게 유지한다.
- 리프레시 토큰을 사용할 경우 저장 위치, 만료, 폐기 정책을 명확히 한다.
- 로그아웃 또는 계정 연결 해제 시 서버가 재사용 가능한 인증 상태를 폐기할 수 있어야 한다.
- 토큰, 인가 코드, OAuth secret은 로그에 남기지 않는다.

## 3. 환경 변수와 설정

- DB 계정, OAuth client secret, AI API key는 `.env` 또는 배포 환경의 secret manager에서 주입한다.
- 민감 설정은 Git에 커밋하지 않는다.
- `application-prod.yaml`에는 운영 기본값만 두고 secret 값은 외부에서 주입한다.
- 개발용 설정과 운영용 설정을 프로필로 분리한다.

## 4. 입력값 검증

- 모든 요청 DTO는 Bean Validation으로 검증한다.
- 지역 코드, 카테고리, 우선순위는 enum으로 제한한다.
- 날짜 범위는 서버에서 검증한다.
- 일정 생성 요청은 최소 콘텐츠 수, 기간, 지역을 검증한 뒤 AI 호출을 수행한다.
- 외부 공유 토큰, 콘텐츠 ID, 일정 ID는 형식과 권한을 모두 확인한다.

## 5. 데이터 신뢰성

AI 일정 생성의 품질은 데이터 신뢰도에 크게 의존한다. 서비스 초기에는 완전 자동 수집보다 검수된 데이터를 중심으로 운영한다.

- 한국관광공사 TourAPI를 1차 데이터 소스로 사용한다.
- 지자체 공식 축제 정보를 보조 데이터로 활용한다.
- 운영시간, 휴무일, 축제 기간, 예약 여부는 별도 필드로 구조화한다.
- 콘텐츠별 데이터 출처와 마지막 검증일을 저장한다.
- 불확실한 정보는 "방문 전 확인 필요" 상태로 관리한다.
- 사용자가 잘못된 정보를 제보할 수 있는 기능을 고려한다.
- AI가 생성한 일정에는 정보 출처와 생성 기준을 함께 제공한다.

### TourAPI 연동 방식

MVP에서 사용하는 공공데이터 포털 API는 다음 2종이다.

- `국문 관광정보 서비스_GW`: 지역 콘텐츠, 행사, 상세정보, 이미지, 코드, 동기화의 1차 원천
- `관광사진 정보_GW`: 콘텐츠 대표 이미지가 부족할 때 보조 이미지 원천

두 API는 모두 공공데이터 포털에서 발급한 동일한 인증키(`PUBLIC_DATA_PORTAL_KEY`)를 사용한다.

- 외부 API 연동은 **OpenFeign RequestInterceptor**를 사용해 인증키를 쿼리 파라미터 `serviceKey`로 주입한다. 헤더가 아님에 주의한다.
- 인증키는 `.env` 파일에 정의하고 `application.yaml`의 `spring.config.import`를 통해 주입한다. Git에 커밋하지 않는다.
- Feign 클라이언트는 `global/client` 패키지에 위치시키고 도메인 코드와 분리한다.
- OpenFeign 타임아웃은 명시적으로 설정하고, 장애 발생 시 `502 Bad Gateway`로 구분해 응답한다.
- 인증키, API 원문 응답, 쿼리 파라미터 전문은 로그에 남기지 않는다.
- 상세 연동 설정은 `.agents/docs/package-manager-guide.md` 참고.

## 6. 데이터 갱신 주기

콘텐츠 유형별로 갱신 주기를 다르게 적용한다. 갱신은 **Spring `@Scheduled` 스케줄러**로 실행한다.

| 데이터 유형 | 갱신 주기 | 비고 |
| --- | --- | --- |
| 기본 정보: 장소명, 위치, 주차 등 | 주 1회 | 매주 월요일 새벽 권장 |
| 운영시간, 휴무일 | 일 1회 | 매일 자정 권장 |
| 축제, 이벤트 | 일 1회 | 축제 시작 1주 전부터 종료일까지 |

- 스케줄러는 `global/scheduler` 패키지에 위치시킨다.
- 갱신 실패 시 기존 저장 데이터를 유지하고 실패 원인을 로그에 남긴다.
- 운영 환경에서는 `@EnableScheduling`을 명시적으로 활성화한다.

## 7. 로그와 모니터링

- 요청 단위 traceId를 생성하고 오류 응답에 포함한다.
- 서버 로그에는 사용자 식별자를 직접 남기기보다 내부 ID 또는 마스킹된 값을 사용한다.
- OAuth 실패, AI 호출 실패, DB 저장 실패는 원인과 외부 제공자 상태를 구분해 기록한다.
- 개인정보, 토큰, 외부 API key, AI 프롬프트 전문은 로그에 남기지 않는다.

## 8. 외부 API 장애 대응

- TourAPI, OAuth, AI 제공자 호출은 타임아웃을 설정한다.
- 외부 API 장애는 `502 Bad Gateway` 또는 재시도 가능한 에러 코드로 구분한다.
- 콘텐츠 조회는 캐시 또는 기존 저장 데이터를 우선 활용할 수 있도록 설계한다.
- AI 일정 생성 실패 시 사용자가 다시 시도할 수 있는 응답을 제공한다.
